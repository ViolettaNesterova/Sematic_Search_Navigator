## Semantic Search Navigator

Проект посвящён исследованию методов семантического 
поиска по программному коду с использованием современных 
моделей эмбеддингов. Основная задача заключается в поиске 
функций и методов по запросам на естественном языке без 
необходимости точного совпадения ключевых слов.

---

## Проблема

Традиционные поисковые механизмы хорошо работают при 
наличии одинаковых терминов в запросе и документе, 
однако часто оказываются неэффективными при 
перефразировании. Например, пользователь может искать 
«проверка срока действия токена», тогда как в коде 
соответствующая логика реализована в функции с 
названием ```verify_jwt_token```.

---

## Решение проблемы

Для решения этой проблемы применяется подход семантического поиска:

1. ****Код и его описание преобразуются в векторные представления.****
2. ****Пользовательский запрос также кодируется в эмбеддинг.****
3. ****Поиск выполняется на основе близости векторов в многомерном пространстве.****
4. ****Пользователю возвращаются наиболее релевантные фрагменты кода.****

---

## Цель исследования

В рамках работы решаются следующие задачи:

- ****построение системы retrieval для программного кода;****
- ****сравнение нескольких мультиязычных embedding-моделей;****
- ****оценка качества поиска с помощью специализированных метрик;****
- ****анализ влияния языка запроса на качество выдачи;****
- ****визуализация структуры эмбеддингов.****

---

## Используемые модели

| Модель                                      | Назначение                                    |
|---------------------------------------------|-----------------------------------------------|
| ```paraphrase-multilingual-MiniLM-L12-v2``` | компактная мультиязычная модель               |
| ```paraphrase-multilingual-mpnet-base-v2``` | более мощная модель для семантического поиска |
| ```intfloat/multilingual-e5-base```         | универсальная модель семейства E5             |

## Метод поиска

Каждая функция из корпуса кодируется в векторное 
пространство. После получения эмбеддинга пользовательского 
запроса вычисляется косинусное сходство между запросом 
и всеми объектами корпуса.

Для оценки результатов используется схема ****Top-3 Retrieval****, 
при которой система возвращает три наиболее близких кандидата.

---

## Метрики качества

### Precision@3

Метрика показывает долю запросов, для которых 
правильный ответ присутствует среди трёх лучших 
результатов поиска.

Формально высокий показатель Precision@3 означает, 
что пользователь с высокой вероятностью увидит нужный 
фрагмент кода без дополнительного поиска.

### Mean Reciprocal Rank (MRR)

MRR учитывает не только наличие правильного ответа, 
но и его позицию в ранжированном списке.

Чем выше релевантный результат расположен в выдаче, 
тем больший вклад он вносит в итоговое значение метрики.

---

## Дополнительный анализ

Помимо основных метрик исследование включает:

- ****сравнение качества для русскоязычных и англоязычных запросов;****
- ****анализ ошибок по предметным категориям;****
- ****изучение сильных и слабых сторон отдельных моделей;****
- ****исследование структуры эмбеддингов после понижения размерности.****

---

## Визуализация пространства эмбеддингов

Для отображения многомерных представлений в двумерном 
пространстве используется алгоритм ****t-SNE****.

Полученные графики позволяют оценить:

- ****степень разделимости тематических категорий;****
- ****плотность кластеров;****
- ****наличие пересечений между различными типами функций;****
- ****качество семантического представления данных.****

## Описание датасета

Датасет предназначен для разработки и оценки системы 
семантического поиска по корпусу программного кода 
(кейс «Навигатор по смыслу», версия 1.0). Он включает 
200 функций на Python и Java, сгруппированных по 
5 тематическим категориям, и 25 тестовых вопросов на 
русском и английском языках.

---

## Структура датасета

```commandline
dataset_v1.0/
├── categories.json       # 5 категорий кода
├── code_corpus.json      # 200 функций (100 Python + 100 Java)
└── eval_questions.json   # 25 тестовых вопросов с эталонными ответами
```

---

## Описание файлов

**code_corpus.jon** - корпус функций для индексирования. Содержит 
200 записей в виде массива JSON.

### Поля каждой функции

| Поле            | Тип    | Описание                                          |
|-----------------|--------|---------------------------------------------------|
| `id`            | string | Уникальный идентификатор: `func_001` … `func_200` |
| `language`      | string | Язык программирования: `python` или `java`        |
| `function_name` | string | Имя функции/метода                                |
| `code`          | string | Исходный код функции                              |
| `description`   | string | Описание назначения функции на русском языке      |
| `category`      | string | Ключ категории из `categories.json`               |

### Распределение по языкам

- `func_001` – `func_100` → `language: "python"`
- `func_101` – `func_200` → `language: "java"`

### Пример записи

```json
{
  "id": "func_001",
  "language": "python",
  "function_name": "verify_jwt_token",
  "code": "def verify_jwt_token(token: str, secret: str) -> dict:\n    \"\"\"Проверяет JWT-токен и возвращает payload или причину невалидности.\"\"\"\n    try:\n        payload = jwt.decode(token, secret, algorithms=[\"HS256\"])\n        return {\"valid\": True, \"data\": payload}\n    except jwt.ExpiredSignatureError:\n        return {\"valid\": False, \"error\": \"expired\"}\n    except jwt.InvalidTokenError:\n        return {\"valid\": False, \"error\": \"invalid\"}",
  "description": "Проверяет JWT-токен и возвращает payload или причину невалидности.",
  "category": "auth"
}
```

---

****eval_questions.json**** - Тестовые вопросы для оценки 
качества retrieval. Содержит 25 записей в виде массива JSON.

### Поля каждого вопроса

| Поле               | Тип    | Описание                                   |
|--------------------|--------|--------------------------------------------|
| `question_id`      | string | Уникальный идентификатор: `q_01` … `q_25`  |
| `query`            | string | Поисковый запрос на естественном языке     |
| `language`         | string | Язык запроса: `ru` или `en`                |
| `correct_chunk_id` | string | ID эталонной функции из `code_corpus.json` |

### Пример записи

```json
{
  "question_id": "q_01",
  "query": "как проверить, истёк ли токен?",
  "language": "ru",
  "correct_chunk_id": "func_001"
}
```

### Категории кода

| Ключ         | Название                     | Цвет      | Описание                                           |
|--------------|------------------------------|-----------|----------------------------------------------------|
| `auth`       | Аутентификация и авторизация | `#E74C3C` | JWT, пароли, сессии, OAuth, 2FA, права доступа     |
| `database`   | Работа с базой данных        | `#3498DB` | CRUD, транзакции, миграции, индексы, bulk-операции |
| `http`       | HTTP-клиенты и API           | `#2ECC71` | REST-вызовы, retry, webhook, upload, кэширование   |
| `validation` | Валидация и парсинг          | `#F39C12` | email, телефон, ИНН, СНИЛС, ISO-даты, URL          |
| `utils`      | Утилиты и хелперы            | `#9B59B6` | форматирование, кэш, slugify, sanitize             |

---

## Что не входит в датасет

- **Нет обучающего (train) набора.** Датасет предназначен исключительно для индексирования корпуса и оценки retrieval-качества.
- **Нет разбивки train/test по функциям.** Все 200 функций индексируются; по 25 вопросам считаются метрики.
- **Нет предвычисленных эмбеддингов.** Выбор модели для построения векторных представлений — на усмотрение команды.
- **Нет ограничений на стек.** Участники свободны в выборе библиотек, моделей и способа хранения индекса.
